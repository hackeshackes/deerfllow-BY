# v1.5.5 灰度公告

> **目的**:把"AI 接入业务系统"的能力以最小风险推进到生产,分 3 段、每段 1 周、共 3 周。
> **决策人**:Tech Lead + Product Owner + SRE
> **回滚 SLA**:任何阶段指标越线 → 30 分钟内回滚到上一阶段

## 阶段 0:内部 1 周(Day 1-7)

**目标**:验证 4 个连接器都能从 MicX → 外部系统单向发送成功,DLQ 工作。

### 参与人
- SRE 1 人(连接器运维)
- 集成工程师 2 人(连接器作者)
- Tech Lead 1 人(审批回滚)

### 触发条件
- v1.5.5 已合并到 main
- tag `v1.5.5` 已发布
- `make up-pro` 在 staging 环境运行无错

### 操作步骤
1. SRE 在 staging 部署 v1.5.5 后,4 个连接器各发 1 条测试消息:
   - 飞书:`feishu-prod` chat_id → 文本 "v1.5.5 灰度测试"
   - 钉钉:`dingtalk-prod` chat_id → 同上
   - 企微:`wecom-prod` user_id → 同上
   - 邮件:`email-prod` to=内部测试邮箱 → subject "v1.5.5 灰度测试"
2. 验证 4 条消息 3s 内到达
3. 故意构造失败场景(删除某连接器一个凭证字段)→ 验证消息进 DLQ 而不是无限重试
4. 在 `/workspace/admin/connectors` 页面查 DLQ 条目,验证可手动 drop

### 验收标准
- [ ] 4 条测试消息均成功投递
- [ ] DLQ 至少捕获 1 条故意失败的消息
- [ ] 失败重试 2 次后正确进 DLQ(不无限重试)
- [ ] admin 页面 DLQ 列表与 drop 按钮工作正常


- 阶段 0 全部验收通过
- 部门负责人签字同意试运行
- CS 跟班 ≥ 2 小时/天答疑

### 操作步骤
1. SRE 在生产部署 v1.5.5(单部门白名单)
2. 部门用户登录 → workspace layout 顶部出现"空间切换器"

## 回滚 Playbook

### 软回滚(单功能关闭)
```bash
# 关闭某个连接器
curl -X POST https://micx.example/api/admin/connectors/disable \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"name": "feishu-prod"}'

# 关闭 SAML
export MICX_SAML_ENABLED=false
kubectl rollout restart deployment/gateway
```

### 硬回滚(整个版本回退)
```bash
# 1. SRE 切流量到旧版本
kubectl set image deployment/gateway gateway=micx-gateway:v1.5.4

# 2. 等待旧版本健康
kubectl rollout status deployment/gateway

# 3. 通知(用上面的"阶段 1 公告"反向发)
邮件标题:[回滚] v1.5.5 已临时回退到 v1.5.4
正文:我们已发现一个影响 [N] 用户的 [P0/P1] 问题,临时回滚。
预计 [时间] 修复后重新上线。
— Tech Lead

# 4. 归档 DLQ(避免回滚后消息丢失)
curl -X POST https://micx.example/api/admin/dlq/drain \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"archive_to": "s3://micx-archive/v1.5.5-dlq-{timestamp}"}'
```

### 回滚时间预算
- 检测到问题 → 决策:≤ 15 分钟
- 软回滚执行:≤ 15 分钟
- 硬回滚执行:≤ 30 分钟
- 公告发出:≤ 30 分钟
- **总 SLA:90 分钟** 从发现到全公司知道
