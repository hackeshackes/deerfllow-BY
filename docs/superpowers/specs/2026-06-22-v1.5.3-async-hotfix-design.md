### 1.2 Verified Root Causes (from container log inspection)

#### Issue 1 — `Event loop is closed` ✅ CONFIRMED REAL

**Location:** `backend/packages/harness/deerflow/agents/middlewares/llm_error_handling_middleware.py:160`

```python
# Line 158-161
if retriable and attempt < self.retry_max_attempts:
    wait_ms = self._build_retry_delay_ms(attempt, exc)
    logger.warning(...)
    time.sleep(wait_ms / 1000)  # ← BLOCKS event loop in async function
```

**Chain of failure:**
1. `awrap_model_call` is `async def`, contains `time.sleep(...)`
2. `time.sleep` blocks the entire asyncio event loop
3. While blocked, other concurrent tasks (e.g. httpx client connection pool, LangGraph internal state) get stuck
4. When `await handler(request)` resumes, the httpx/openai client's internal state has been affected by another thread/task, causing `RuntimeError: Event loop is closed`

**Container log evidence:**
```
[error] LLM call failed after 1 attempt(s): Event loop is closed
[deerflow.agents.middlewares.llm_error_handling_middleware]
Traceback: File ".../llm_error_handling_middleware.py", line 180, in awrap_model_call
  return await handler(request)
RuntimeError: Event loop is closed
```

**Similar `time.sleep` in async context — also need fixing:**
- `backend/packages/harness/deerflow/agents/memory/queue.py:160`
- `backend/packages/harness/deerflow/models/claude_provider.py:296,303`
- `backend/packages/harness/deerflow/models/openai_codex_provider.py:208`

#### Issue 2 — `Permission denied: /mnt/user-data` ⚠️ NEEDS INVESTIGATION

**User report:** Permission denied when reading/writing `/mnt/user-data`.

**Initial inspection:** Container has `/mnt/` empty (no mounts). The path `/mnt/user-data` is a **virtual path** that maps to actual filesystem location per-thread.

**Path translation logic:** `backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py`
- Reserved prefixes: `["/mnt/acp-workspace", "/mnt/user-data"]`
- Maps to: `backend/.deer-flow/threads/{thread_id}/user-data/`

**Initial verification:** Container at `/app/backend/.deer-flow/threads/{thread_id}/user-data/` shows `drwxr-xr-x 5 root root` — directory exists and is writable by root (which is what container runs as).

**No actual `PermissionError` or `EACCES` found in container logs.**

**Status:** Need to **reproduce the issue locally** to confirm root cause. Could be:
- (a) Sandbox subprocess running with different user (uid mismatch)
- (b) Specific thread ID has different ownership (was created by different process)
- (c) Specific sub-path (e.g. `/mnt/user-data/outputs`) has different permissions
- (d) Race condition with directory creation
- (e) User reported issue is actually a misreport

#### Issue 3 — `BlockingError: os.mkdir` ✅ CONFIRMED REAL

**Location:** `backend/packages/harness/deerflow/agents/memory/storage.py:152` (sync I/O inside async LangGraph context)

```python
def save(self, memory_data, agent_name=None, user_id=None) -> bool:
    file_path = self._get_memory_file_path(agent_name, user_id)
    cache_key = (agent_name, user_id)
    try:
        file_path.parent.mkdir(parents=True, exist_ok=True)  # ← sync I/O
        memory_data["lastUpdated"] = utc_now_iso_z()
        temp_path = file_path.with_suffix(".tmp")
        with open(temp_path, "w", encoding="utf-8") as f:  # ← sync I/O
            json.dump(memory_data, f, indent=2, ensure_ascii=False)
        temp_path.replace(file_path)
    except OSError as exc:
        ...
```

**Container log evidence:**
```
[error] Memory update failed: Blocking call to os.mkdir
[Heads up] LangGraph dev identified a synchronous blocking call in your code.
When running in an ASGI web server, blocking calls can degrade performance
for everyone since they tie up the event loop.
```

**LangGraph uses `blockbuster` library to detect sync blocking calls in async context.** MemoryMiddleware calls `MemoryUpdater.update()` (sync) which calls `MemoryStorage.save()` (sync file I/O).

**Call chain:**
```
memory_middleware.py → MemoryUpdater.update() [SYNC]
  → MemoryStorage.save() [SYNC, has os.mkdir + open + json.dump]
  → blockbuster detects → BlockingError
```

## 3. Fix Strategy

### 3.1 P0 Fix — Replace `time.sleep` with `asyncio.sleep` in async contexts

**Pattern:**
```python
# BEFORE (in async function)
def awrap_model_call(self, ...):
    ...
    time.sleep(wait_ms / 1000)  # WRONG

# AFTER
async def awrap_model_call(self, ...):
    ...
    await asyncio.sleep(wait_ms / 1000)  # CORRECT
```

**Files modified:**
1. `llm_error_handling_middleware.py` — already async, just swap sleep
2. `memory/queue.py` — already in async context (background worker), swap sleep
3. `models/claude_provider.py` — check if in async context, use `asyncio.sleep` or `asyncio.to_thread(time.sleep, X)`
4. `models/openai_codex_provider.py` — same as claude_provider

**Tests added:**
- `tests/test_async_no_blocking_in_retry.py` — Patch `asyncio.sleep`, verify it's called instead of `time.sleep` in LLM retry middleware
- `tests/test_async_no_blocking_in_memory_queue.py` — Verify memory queue uses asyncio.sleep
- `tests/test_async_no_blocking_in_model_providers.py` — Verify claude/openai providers use asyncio.sleep or asyncio.to_thread

### 3.2 P1 Fix — Make MemoryUpdater async

**Pattern:**
```python
# BEFORE
class MemoryStorage:
    def save(self, ...) -> bool:
        file_path.parent.mkdir(parents=True, exist_ok=True)
        with open(file_path, "w") as f:
            json.dump(...)
        return True

# AFTER
class MemoryStorage:
    async def save(self, ...) -> bool:
        await asyncio.to_thread(self._sync_save, ...)

    def _sync_save(self, ...) -> bool:
        # All the existing sync logic here
        ...

class MemoryUpdater:
    async def update(self, ...) -> bool:
        return await self._storage.save(...)
```

**Files modified:**
1. `memory/storage.py` — split save() into async wrapper + sync `_sync_save()`
2. `memory/updater.py` — make `MemoryUpdater.update()` async
3. `memory/queue.py` — change background thread to `asyncio.create_task()`

**Tests added:**
- `tests/test_memory_storage_async.py` — Verify save() doesn't block event loop
- `tests/test_memory_updater_async.py` — Verify update() is async + non-blocking
- `tests/test_memory_queue_uses_asyncio.py` — Verify queue uses asyncio.create_task

### 3.3 P2 Fix — Investigate and fix `/mnt/user-data` permission

**Investigation plan:**
1. Reproduce in worktree by running uvicorn with sandbox command
2. Inspect actual thread directory permissions:
   ```bash
   ls -la backend/.deer-flow/threads/{thread_id}/user-data/
   find backend/.deer-flow/threads/{thread_id}/user-data/ -uid -user
   ```
3. Trace path translation in `local_sandbox_provider.py` to find where path resolution fails
4. Common root causes:
   - **Different uid** when sandbox subprocess runs vs container main process
   - **Missing parent directory** in path translation
   - **Read-only mount** in docker-compose.yaml (unlikely since `/mnt/` is empty)

**Likely fix:**
- If uid mismatch: ensure subprocess runs with same uid as container main process
- If missing parent directory: ensure `mkdir -p` is called for parent before write
- If sandbox provider bug: fix path translation to handle nested non-existent paths

**Tests added:**
- `tests/test_local_sandbox_user_data_write.py` — Verify user can write to `/mnt/user-data/foo.txt`
- `tests/test_local_sandbox_user_data_nested.py` — Verify nested paths like `/mnt/user-data/workspace/foo/bar.txt` work

## 6. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Async refactor breaks existing sync callers | Add test that calls `MemoryUpdater.update()` from sync code → expect graceful failure or sync wrapper |
| P2 root cause is deep (filesystem permissions) | Mark P2 as "investigate; if too deep, defer to v1.5.4 hotfix 2" |
| Rebuild takes 15 min and may fail | Have rollback plan: `docker tag docker-gateway:backup` before rebuild |
| Hotfix conflicts with v1.5.4 merge | Hotfix is a separate branch from v1.5.4 → no conflict if applied to v1.5.3 first |
| `asyncio.to_thread` adds overhead | Acceptable for memory saves (low-frequency) |
| New tests might be flaky (timing) | Use generous timeouts (100ms instead of 1ms) |
