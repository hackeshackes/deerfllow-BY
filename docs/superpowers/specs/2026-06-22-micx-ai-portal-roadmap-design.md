## 1. 概述与目标

### 1.1 产品定位(北星)

> **MicX 是企业私有部署的 AI 门户 + 内部团队协作平台 + 各项任务/业务系统的统一入口。**
> 员工的所有 AI 交互、跨部门协作、外部系统调度,都通过 MicX 完成。

### 1.2 这个版本序列要回答的问题

把 MicX 从"功能集合"演进为"企业 AI 操作系统",**6 个月内**(v1.5.4 / v1.5.5 / v1.6.0 三个版本)完成从"全公司 IT 批准备案"到"业务部门日常使用"的完整闭环。

### 1.3 为什么是 3 个版本而不是 6 个

原方案 A 计划了 v1.6.0 → v2.1.0 共 6 个大版本(约 12 个月跨度)。**压缩为 3 个连续小步**:
- 与 MicX 历史上"每 2-4 周一个版本"的发布节奏一致
- 5+ 人团队能在 3 条版本线上保持稳定节奏,避免大版本延期
- 每个版本都有可验证的产出,降低"做大版本做到一半推翻"的风险
- 后续 v1.7+ 版本可在 v1.6.0 之后视情况再开

### 1.4 战略校准:不做什么

- **不做**:通用 AI Agent marketplace(企业内不需要,反而增加风险)
- **不做**:实时多人在线协作(Cursor 风格光标)— Y.js CRDT 是坑,等 v1.7+
- **不做**:支持 10+ 模型供应商的"模型聚合器" — 锁定 3-5 个主模型 + 5 个备选
- **不做**:全员私有模型微调 — 投入产出比不划算,等 12 月后再评估
- **不做**:移动 App — Web 响应式适配足够
- **不做**:完整 GDPR/CCPA 数据生命周期 — 等客户有合规需求时再说

## 4. 三个版本的详细范围

### 4.1 v1.5.4 — 企业身份与权限地基(5-6 周)

**版本目标**:让 IT 部门批 MicX 进入全公司部署

#### 范围

| 模块 | 内容 | 优先级 |
|---|---|---|
| **OIDC 集成** | Keycloak / Okta / Azure AD / Authing 适配,PKCE 流程,refresh token,租户隔离 | P0 |
| **RBAC v2** | 三层模型:角色(系统/部门/项目) → 资源(workspace/thread/knowledge/agent) → 操作 | P0 |
| **审计 v2** | 全操作审计(对话/上传/配置/技能启用/IM 消息),不可变日志,导出/对接 SIEM | P0 |
| **SCIM 2.0** | 从 IdP 自动同步组织架构(用户/部门/项目变化) | P1 |
| **Admin 治理控制台** | 部门/项目树形管理、SSO 配置 UI、审计查询 UI | P0 |

#### 验收标准

- ✅ 能在生产环境启用 OIDC 登录(至少 3 个 IdP 完成端到端验证)
- ✅ 审计日志覆盖 100% 写操作
- ✅ SCIM 同步组织架构正确性 ≥ 99%
- ✅ RBAC v2 单元测试覆盖率 ≥ 90%

#### 不做(明确推迟)

- SAML(推到 v1.5.5)
- 细粒度 ABAC(推到 v1.6.0)
- 移动端 SSO 适配(暂不需要)
- 多因子认证(先依赖 IdP 端 MFA)

#### 关键风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| IdP 集成测试覆盖度难保证 | 高 | 高 | 准备"5 分钟接入 Keycloak/Okta/Authing"参考文档 + 沙箱 IdP 测试套件 |
| 审计日志性能开销 | 中 | 中 | 异步批量写,PostgreSQL append-only 表 + 定期归档 |
| RBAC v2 与现有 v1 RBAC 兼容 | 中 | 中 | 双轨运行,新资源走 v2,旧资源迁移清单 |

### 4.2 v1.5.5 — 业务连接器 + 协作闭环(6-7 周,双线并行)

**版本目标**:让业务部门立刻看到"AI 接入业务系统的价值",同时兑现 v1.4.4 路线图核心部分

> 这是**双线并行**的版本。集成线和协作线同时推进,互不阻塞。

#### 集成线(主战场,4 个连接器)

| 模块 | 内容 | 优先级 |
|---|---|---|
| **连接器运行时** | 统一 `Connector` 协议(YAML 声明式配置) | P0 |
| **飞书连接器(补全)** | 群消息 + 日历 + 审批,从 v1.5.3 的工具升级为连接器 | P0 |
| **钉钉连接器(补全)** | 审批 + 群消息 + 机器人,补全 v1.5.3 已有能力 | P0 |
| **企微连接器(补全)** | 群消息 + 应用消息 + 通讯录 | P0 |
| **邮件连接器(新增)** | IMAP 收件 + SMTP 发件 + Exchange 兼容 | P0 |
| **Webhook 双向桥** | 外部事件 → MicX 触发;MicX 输出 → 外部系统 | P0 |
| **失败重试 + 死信队列** | 投递失败 3 次入死信,管理员手动重放 | P0 |
| **SAML 支持** | v1.5.4 推后的 SAML 补回 | P1 |

> 钉钉/企微/邮件在 v1.5.3 已有基础,这里是**升级为统一连接器运行时**而非从零做。

#### 协作线(兑现 v1.4.4 路线图核心)

| 模块 | 内容 | 优先级 |
|---|---|---|
| **一级空间切换** | 顶栏"个人/项目/部门"空间切换,所有列表跟随过滤 | P0 |
| **对话分区** | Recent / 自动化结果 / 外部渠道对话 分区显示 | P0 |
| **新建对话场景化** | 选空间 → 选知识库 → 选工作流 → 启动 | P1 |
| **@提及 + 订阅** | @部门、@项目、@人;订阅工作流/知识库/对话 | P1 |
| **跨部门发布** | 个人对话一键发布为部门可见,保留来源链接 | P2 |

#### 验收标准

- ✅ 4 个连接器端到端通过(模拟外部事件 → MicX 触发 → 输出回原系统)
- ✅ 用户能回答 v1.4.4 路线图那 5 个问题(在哪个空间 / 谁能看到 / 参考了什么 / 自动化结果去哪 / 工作流归谁)
- ✅ 投递延迟 < 3s,失败重试生效
- ✅ SAML 接入至少 2 个企业 IdP(例如 ADFS、Shibboleth)

#### 不做(明确推迟)

- Jira/Linear 工单连接器(推到 v1.6.0)
- Slack 连接器(推到 v1.6.0)
- 业务部门自建连接器(推到 v1.7+)
- 共享对话评论(推到 v1.6.0)
- 跨部门发布(推到 v1.6.0)

#### 关键风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 双线并行扯皮 | 中 | 中 | 集成线 owner 是后端集成负责人,协作线 owner 是前端负责人,每周一对齐 |

## 5. 架构演进

### 5.1 当前架构(v1.5.3)

```
Nginx (2026) → LangGraph (2024) + Gateway FastAPI (8001) + Next.js 前端
                  ↓
              工具/Skills/MCP/Memory/上传/工单/审计/IM 渠道(都在一个 Gateway 内)
```

**问题**:Gateway 是单体的、所有功能耦合、SSO 不存在、审计只覆盖配置变更。

### 5.2 目标架构(v1.6.0 末)

```
                           ┌─ 企业 IdP (OIDC/SAML)
                           │
Nginx (TLS 1.3, WAF) ─────┴─ MicX Gateway
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   Identity/SSO          AI Runtime             协作/空间(v1.5.5)
        │                     │                     │
   OIDC/SAML(v1.5.4/5)   Model Router (v1.6)   工作区服务
   RBAC/ABAC             Prompt Registry       共享权限
   Audit Stream (v1.5.4) Tool Registry         评论/订阅(v1.5.5/v1.6)
   SCIM (v1.5.4)         Agent Runtime         通知
        │                     │                     │
   Connectors(v1.5.5)        │                     │
   飞书/钉钉/企微/邮件        │                     │
        │                     │                     │
        └─────────┬───────────┴─────────────────────┘
                  │
            Workflow Engine (v1.6 画布骨架)
```

### 5.3 关键架构原则(不变量)

1. **数据平面与控制平面分离** — 用户操作走数据平面,管理操作走控制平面
2. **所有写操作可审计** — v1.5.4 起强制
3. **任何"非自己部门的资源"必须显式授权** — RBAC v2 + ABAC 共同保证
4. **连接器统一协议** — 任何外部系统接入走 `Connector` 接口,避免特殊实现堆叠
5. **可观测性优先** — OpenTelemetry 全面接入,任何故障可定位到具体 span
