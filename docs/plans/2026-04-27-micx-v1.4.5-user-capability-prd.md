# MicX v1.4.5 PRD - 用户能力增强：沉淀与复用

**版本**: v1.4.5  
**主题**: 用户能力增强：沉淀与复用  
**目标**: 让用户把一次有效对话，低成本转化为可保存的资料、可共享的团队知识、可复用的工作流、可持续执行的自动化。  
**日期**: 2026-04-27  
**状态**: 草案，待用户确认

---

## 一、版本判断

v1.4.4 已经把 MicX 从“功能菜单集合”推进到“空间化用户工作台”。用户现在更容易知道自己在哪个空间、对话属于哪里、资料库和自动化从哪里进入。

v1.4.5 不应继续堆更多入口。上一版已经证明，过重的新建页引导会打断用户直接开始工作的节奏。ChatStartPanel 已暂时隐藏，说明 v1.4.5 的增强应更多发生在对话完成后、回复旁边、线程级动作里。

v1.4.5 的核心判断是：

> v1.4.4 回答“我在哪里做事”。  
> v1.4.5 回答“做完之后，价值怎么留下来、复用起来、继续自动跑”。

本版本的主轴是 **Capture → Save → Reuse → Automate**：

1. Capture：识别一次对话里的有效结果。
2. Save：把结果保存成资料或总结。
3. Reuse：把做法沉淀成工作流模板。
4. Automate：把可重复做法转成自动化草稿。

## 二、目标用户

### 2.1 普通成员

典型行为：

- 用 AI 分析文件、写文档、总结会议、生成方案。
- 在个人空间里探索，再决定是否共享给团队。
- 经常重复类似任务，但每次都重新写 prompt。

痛点：

- 好答案留在聊天里，过几天找不到。
- 想把结论给团队复用，但不知道该存到哪里。
- 每次做相同任务都要重新描述背景、格式、要求。
- 不清楚 AI 引用的资料是否可靠、是否已处理完成。

### 2.2 团队负责人 / 空间管理员

典型行为：

- 希望团队成员把高质量分析沉淀为共享资料。
- 希望把团队常用做法固化为工作流。
- 希望把日报、周报、风险扫描等固定动作变成自动化。

痛点：

- 团队知识分散在个人聊天中。
- 优秀 prompt 和分析流程靠人记忆传播。
- 自动化创建门槛仍偏配置化，不够从真实对话自然转化。

### 2.3 系统管理员

v1.4.5 不重点增强系统管理员。Admin 后台只需要支持必要的资料、工作流、自动化权限边界，不扩展新的管理中心。

## 三、版本目标

### 3.1 业务目标

1. 提高用户“完成一次任务后继续复用”的比例。
2. 降低从对话到知识沉淀、工作流、自动化的操作成本。
3. 强化个人空间到共享空间的显式发布链路，避免误共享。
4. 让团队能积累可复用资产，而不是只积累聊天记录。

### 3.2 用户结果

v1.4.5 完成后，用户应能回答：

1. 这次对话有哪些内容值得留下？
2. 我能不能一键把它存成个人或团队可复用资料？
3. 下次做类似事情，能不能直接复用这次做法？
4. 这个做法能不能转成自动化持续执行？
5. 我引用的资料是否已经处理完成、是否值得信任？

## 四、范围

### 4.1 本版本必须交付

1. **对话级“沉淀”动作栏**：在已有对话顶部或导出菜单旁增加沉淀入口。
2. **一键生成对话总结**：把当前线程归纳为结构化总结草稿。
3. **一键保存到资料库**：把总结或选中 AI 回复保存为资料库文档。
4. **发布到共享空间**：将私有对话总结显式发布到当前共享空间资料库或共享对话。
5. **从对话生成工作流草稿**：把一次成功做法转成可编辑的工作流 prompt 模板。
6. **从对话/工作流生成自动化草稿**：带入任务描述、频率、输出位置，跳转自动化创建页。
7. **资料质量状态可见**：资料库文档显示处理状态、chunk 数、token 数、失败/待处理状态。

### 4.2 本版本可选交付

1. 工作流轻量参数化，如 `{主题}`、`{周期}`、`{输出格式}`。
2. 保存资料时重复标题提醒。
3. 引用来源预览增强。
4. 线程总结历史版本。

### 4.3 本版本不做

1. 不做复杂工作流可视化编排器。
2. 不做实时多人协作编辑。
3. 不做通知订阅中心。
4. 不重构向量检索底层。
5. 不新增外部平台大扩展。
6. 不重新设计新建对话首页大面板。

## 五、核心用户链路

### 5.1 从一次对话沉淀为资料

```text
用户完成对话
→ 点击“沉淀”
→ 选择“生成总结”或“保存当前回复”
→ 选择资料库：个人 / 当前空间 / 全局（仅管理员）
→ 编辑标题和摘要
→ 保存
→ 资料库页面出现新文档，显示处理中 / 已完成
```

验收：

- 用户不用导出 Markdown 再手动上传。
- 保存前明确显示目标空间和可见范围。
- 保存后有 toast 和跳转入口。

### 5.2 从一次成功对话生成工作流

```text
用户认为这次任务可复用
→ 点击“生成工作流草稿”
→ 系统提取目标、输入、步骤、输出格式
→ 用户编辑名称、描述、prompt 模板
→ 保存为个人工作流或共享工作流草稿
```

验收：

- 工作流草稿不是立即公开，必须用户确认保存。
- 普通用户默认保存为个人工作流。
- 空间管理员可选择共享到当前空间。

### 5.3 从工作流生成自动化

```text
用户已有工作流
→ 点击“转为自动化”
→ 自动化创建页预填 prompt、输出位置、可见范围
→ 用户选择频率
→ 保存自动化任务
```

验收：

- 自动化创建页支持 URL 参数或 draft state 预填。
- 自动化结果继续进入“自动化结果”分区。
- 输出位置明确显示：私有对话 / 当前空间对话 / 指定资料库。

## 六、功能需求

### 6.1 对话沉淀入口

入口位置：

- 优先放在对话顶部右侧工具区，靠近 `ExportTrigger`。
- 不放在新建对话首页，不再引入大面积引导卡片。

建议文案：

| 中文 | 英文 | 行为 |
|------|------|------|
| 沉淀 | Capture | 打开沉淀菜单 |
| 生成总结 | Summarize | 生成当前对话总结草稿 |
| 保存为资料 | Save to Sources | 保存总结或回复到资料库 |
| 生成工作流 | Create Workflow Draft | 抽取可复用模板 |
| 转为自动化 | Create Automation Draft | 跳转自动化创建页并预填 |

验收：

- 空线程不显示沉淀入口。
- 加载中线程禁用会修改内容的动作。
- 所有动作都有成功/失败提示。

### 6.2 对话总结草稿

总结结构：

```markdown
# 标题

## 背景
## 关键结论
## 决策 / 建议
## 可复用步骤
## 后续动作
```

第一阶段实现策略：

- 优先使用已有线程消息在前端组合为 Markdown 草稿。
- 如果后端已有 summary 能力，则接入后端；没有则先做“保存原始结构化导出”。
- 不强行引入新的 LLM summarization endpoint，除非现有能力可以复用。

验收：

- 总结可编辑。
- 可复制。
- 可保存到资料库。

### 6.3 保存到资料库

保存对象：

1. 当前 AI 回复。
2. 当前线程总结。
3. 完整线程 Markdown。

目标资料库选择：

| 用户身份 | 可选目标 |
|----------|----------|
| 普通成员 | 私有资料库、当前空间资料库 |
| 空间管理员 | 私有资料库、当前空间资料库 |
| owner/admin | 私有资料库、当前空间资料库、全局资料库 |

后端需求：

- 如果现有 `/api/knowledge/{id}/documents` 只支持文件上传，则新增文本保存接口。
- 建议接口：`POST /api/knowledge/{id}/documents/text`

请求示例：

```json
{
  "title": "项目周报生成流程总结",
  "content": "Markdown content",
  "source_type": "thread_summary",
  "source_thread_id": "thread_xxx",
  "source_message_id": "optional"
}
```

验收：

- 保存后生成知识库文档。
- 文档 metadata 记录来源 thread。
- 文档处理状态可见。

### 6.4 发布到共享空间

发布不是默认行为，必须显式确认。

发布方式：

1. 将私有线程 visibility 改为 workspace。
2. 将线程总结保存到当前空间资料库。
3. 两者可分开操作，避免用户误解。

确认弹窗必须显示：

- 当前空间名称。
- 发布后谁能看到。
- 发布内容类型：对话 / 总结 / 资料。

验收：

- 个人空间内容不会自动共享。
- 发布后 recent chat badge 显示“已共享”。
- 非 owner 不能修改不属于自己的私有线程。

### 6.5 生成工作流草稿

工作流草稿字段：

```json
{
  "name": "string",
  "description": "string",
  "prompt_template": "string",
  "recommended_mode": "chat | research | deep_research",
  "default_visibility": "private | workspace",
  "source_thread_id": "string"
}
```

第一阶段可以不新增复杂 DSL。只做 prompt template。

验收：

- 从对话能进入工作流草稿页或弹窗。
- 用户保存前能编辑名称、说明、prompt。
- 保存后 `/workspace/workflows` 可看到。

### 6.6 生成自动化草稿

自动化草稿字段：

```json
{
  "title": "string",
  "prompt": "string",
  "schedule_type": "cron | interval",
  "visibility": "private | workspace",
  "output_target": "thread | knowledge_base",
  "source_thread_id": "optional",
  "source_workflow_id": "optional"
}
```

验收：

- `/workspace/automations/new` 支持预填。
- 用户仍需手动确认频率和保存。
- 创建后结果进入自动化结果分区。

### 6.7 资料质量状态

在资料库页面和资料详情页显示：

| 状态 | 含义 | 用户动作 |
|------|------|----------|
| processing | 正在解析 | 等待或刷新 |
| ready | 可检索 | 可用于 AI 上下文 |
| failed | 解析失败 | 重试 / 删除 |
| stale | 可能过期 | 重新上传或确认 |

第一阶段优先显示已有字段：

- `status`
- `chunk_count`
- `token_count`
- `processed_at`

验收：

- 用户能看到资料是否可用。
- 失败资料有明确状态，不静默消失。

## 七、权限与安全

### 7.1 可见性原则

1. 私有对话和私有资料默认仅创建者可见。
2. 发布到共享空间必须显式确认。
3. 全局资料仅 owner/admin 可创建或修改。
4. 从对话生成工作流默认私有。
5. 从工作流生成自动化时沿用当前空间上下文，但仍需确认。

### 7.2 防误操作

高风险动作必须二次确认：

- 发布对话到共享空间。
- 保存资料到共享空间。
- 创建共享工作流。
- 创建共享自动化。

## 八、信息架构影响

### 8.1 不新增一级导航

v1.4.5 不新增一级菜单。复用现有：

- 对话
- 资料库
- 自动化
- 工作流

新增能力应嵌入已有页面：

- 对话页：沉淀入口。
- 资料库页：状态和来源显示。
- 工作流页：草稿和来源显示。
- 自动化创建页：预填能力。

### 8.2 ChatStartPanel 暂不恢复

新建对话页保持轻量，只显示 Welcome + 输入框。

原因：

- 用户已明确反馈该区域不美观。
- v1.4.5 核心是“完成后沉淀”，不是“开始前引导”。
- 入口越多，用户越不直接开始。

## 九、验收标准

### 9.1 功能验收

- 已完成对话能打开“沉淀”菜单。
- 能生成可编辑总结草稿。
- 能将总结保存到资料库。
- 能将私有内容显式发布到共享空间。
- 能从对话生成工作流草稿。
- 能从对话或工作流生成自动化草稿。
- 资料库文档状态清晰可见。

### 9.2 体验验收

- 不增加新建对话页面复杂度。
- 用户不用离开对话页即可开始沉淀。
- 发布到团队空间前必须看到确认信息。
- badge 和状态不堆叠，不遮挡标题。

### 9.3 技术验收

```bash
cd frontend && pnpm lint && pnpm typecheck
cd frontend && BETTER_AUTH_SECRET=local-dev-secret pnpm build
cd backend && make lint
```

部署后浏览器验证：

- `/workspace/chats/new`
- `/workspace/chats/{thread_id}`
- `/workspace/knowledge`
- `/workspace/workflows`
- `/workspace/automations/new`

## 十一、风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 保存资料需要后端文本接口 | 前端动作无法闭环 | 第一阶段新增最小文本保存 API |
| 工作流草稿模型不稳定 | 范围膨胀 | 先做 prompt template，不做 DSL |
| 总结质量依赖 LLM | 可能变慢或不稳定 | 先支持可编辑草稿和完整 Markdown 保存 |
| 用户误发布 | 数据泄漏 | 所有共享动作显式确认 |
| UI 又变复杂 | 重复 v1.4.4 ChatStartPanel 问题 | 动作放在对话完成后，不放起始页 |

---

## 十二、版本结论

v1.4.5 不做更多入口，不做更大的后台，不做复杂知识工程。

它只做一件事：

> 让用户已经完成的一次好工作，变成可以留下、共享、复用、自动执行的资产。

这是 MicX 从“AI 对话工具”走向“团队智能工作台”的关键一步。
