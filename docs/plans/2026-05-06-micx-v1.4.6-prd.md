# MicX v1.4.6 PRD - 可信知识与记忆

**版本**: v1.4.6
**主题**: 可信知识与记忆
**目标**: 让用户知道 AI 用了什么资料、记住了什么信息、是否可信；用户可以查看、纠正和管理自己的记忆。
**日期**: 2026-05-06
**状态**: 草案，待用户确认

---

## 一、版本判断

v1.4.5 完成了"沉淀与复用闭环"——用户可以把对话变成资料、工作流、自动化。

v1.4.6 要回答的问题是：

> **用户能不能相信 AI 的回答是基于正确资料的？**

现在用户有两个具体痛点：

1. **不知道 AI 用了什么资料** — AI 回复时没有显示引用来源，用户无法判断答案是否基于可信资料。

2. **不知道记忆里有什么** — 记忆是 AI 自动写入的，用户看不到里面记了什么对不对，无法纠正错误记忆。

v1.4.6 的核心判断：

> v1.4.5 解决了"做完之后价值怎么留下"。v1.4.6 解决"留下的东西是否可信、是否正确"。

本版本的主轴是 **Visible Sources → Trustable Memory → Correctable Facts**：

1. Visible Sources：AI 回答时显示引用了哪些资料。
2. Trustable Memory：用户能查看自己的记忆内容。
3. Correctable Memory：用户可以删除错误记忆、补充正确事实。

## 二、目标用户

### 2.1 普通成员

典型行为：

- 上传了一份资料，用 AI 分析它。
- 问了一个需要参考文档的问题。
- AI 自动记住了一些关于用户的事实。

痛点：

- 不确定 AI 的回答是不是真的用了那份资料。
- 不知道记忆里记了什么对不对。
- 发现了错误记忆，不知道怎么改。

### 2.2 团队负责人 / 空间管理员

典型行为：

- 上传团队文档，希望 AI 基于它回答。
- 监控团队成员是否正确使用知识库。

痛点：

- 无法确认 AI 是否真的引用了团队资料。
- 希望能看到成员的记忆使用情况（只读）。

### 2.3 系统管理员

v1.4.6 增强系统管理员的记忆查看能力，但不能修改。

## 三、版本目标

### 3.1 业务目标

1. 提高用户对 AI 回答的信任度。
2. 让用户对自己的记忆有完全的可控权。
3. 让团队负责人能看到成员记忆使用情况（只读），用于监督和培训。

### 3.2 用户结果

v1.4.6 完成后，用户应能回答：

1. AI 回答这个问题时引用了哪份资料？
2. 那份资料的状态是什么？（处理中 / 已完成 / 失败）
3. AI 关于我的记忆里记了哪些事实？
4. 我能不能删除一个错误记忆？
5. 我能不能手动添加一个正确事实？

## 四、范围

### 4.1 本版本必须交付

1. **AI 回答引用来源展示**：在 AI 回复旁显示引用了哪些知识库文档。
2. **引用文档状态**：显示被引用的文档是否可用（处理中 / 已完成）。
3. **用户记忆中心**：用户可以查看自己的记忆摘要、fact 列表、来源和更新时间。
4. **记忆删除**：用户可以删除任意一条错误记忆。
5. **手动添加记忆**：用户可以手动添加一条正确事实。
6. **Admin 记忆查看**：space owner 可以查看空间内所有用户的记忆（只读）。

### 4.2 本版本可选交付

1. 引用片段预览：点击引用时显示文档中的相关段落。
2. 记忆编辑：用户可以编辑已有 fact 的内容和分类。
3. 知识库失败文档重试：用户可以主动触发失败文档的重新处理。
4. 重复标题提醒增强。

### 4.3 本版本不做

1. 不做知识图谱或复杂关系可视化。
2. 不做记忆自动冲突合并。
3. 不做多人同时编辑同一份记忆。
4. 不重构 RAG 底层检索算法。
5. 不做团队记忆画像或趋势分析。

## 五、核心用户链路

### 5.1 AI 回答显示引用来源

```text
用户提问
→ AI 分析问题，检索知识库
→ AI 生成回答
→ 回答旁显示引用 badge（1 或多个）
→ 用户点击 badge
→ 显示引用文档列表（名称 + 状态）
```

验收：

- 引用 badge 在 AI 回复区域内，不打断阅读流。
- 用户能看到引用的文档名称和状态。
- 点击可展开引用文档详情。

### 5.2 用户查看自己的记忆

```text
用户进入设置 / 记忆页面
→ 看到记忆总览（事实数、更新时间）
→ 可按类别筛选（context / preference / work 等）
→ 每个 fact 显示：内容、来源 thread、创建时间
→ 可删除、可手动添加
```

验收：

- 用户不需要技术知识就能看懂记忆内容。
- 删除不需要二次确认（Toast 撤回）。
- 手动添加需要填写内容 + 确认。

### 5.3 Admin 查看空间成员记忆

```text
Admin 进入工作区管理
→ 选择成员
→ 查看该成员的记忆摘要（只读）
→ 不能修改、不能删除
```

验收：

- Admin 只能看，不能写。
- 空间 owner 可以看到空间内所有用户的记忆。
- 普通成员不能看到其他人的记忆。

## 六、功能需求

### 6.1 AI 回答引用来源展示

#### 前端需求

在 AI 消息组件中：

- 如果本条消息涉及知识检索，显示引用来源 badge。
- Badge 样式：`📄 引用了 2 份资料`，点击展开文档列表。
- 文档列表显示：`文档名` + 状态 badge（处理中 / 已完成 / 失败）。

#### 实现位置

- `frontend/src/components/workspace/messages/` 下的 AI 消息渲染组件。
- 复用现有的 `source_documents` 或类似字段（如果存在）。
- 如果不存在，需要后端在 AI 响应 metadata 中注入引用列表。

#### 后端需求

Agent 回复时需要在 metadata 中包含：

```json
{
  "citations": [
    {
      "kb_id": "kb_xxx",
      "doc_id": "doc_yyy",
      "doc_name": "项目文档.pdf",
      "chunk_content": "相关段落...",
      "similarity_score": 0.85
    }
  ]
}
```

#### 验收

- 有引用时显示 badge。
- 无引用时正常显示，不显示空 badge。
- 引用列表可点击展开/收起。

### 6.2 用户记忆中心

#### 已有基础

`frontend/src/components/workspace/settings/memory-settings-page.tsx` 已存在，包含：

- 记忆总览（context sections + facts）。
- 导出 / 导入。
- 手动添加 fact。

#### 需要增强

1. **Summary 视图**：把记忆按 category 分组显示。
2. **来源显示**：每个 fact 显示来源 thread，用户可跳转。
3. **删除功能**：每个 fact 可删除，不需要二次确认。
4. **手动添加**：保留现有添加 dialog。

#### 已有 API

- `GET /api/memory` — 获取记忆数据（已实现）
- `POST /api/memory/facts` — 创建 fact（已实现）
- `DELETE /api/memory/facts/{fact_id}` — 删除 fact（已实现）
- `PATCH /api/memory/facts/{fact_id}` — 更新 fact（已实现）

#### 验收

- 用户能在设置页面看到完整的记忆视图。
- 可以按类别筛选。
- 可以删除任意 fact。
- 可以手动添加 fact。

### 6.3 Admin 记忆查看

#### 后端新增

`GET /api/admin/memory/{user_id}` — 获取指定用户的记忆（仅 admin / space owner）

需要在 router 中：

1. 校验当前用户是否有权限（ admin 或 space owner）。
2. 调用 `get_memory_data()` 时传入目标 user_id（需要改造现有函数）。

#### 前端新增

`frontend/src/app/workspace/admin/users/[user_id]/memory/page.tsx`（或作为 user detail 的一部分）

显示：

- 该用户的记忆总览。
- Fact 列表（只读）。
- 不能编辑、删除。

#### 验收

- Admin / space owner 可以查看空间内任意用户的记忆。
- 普通成员不能访问该 API 和页面。
- 查看是只读的，没有编辑入口。

### 6.4 知识库文档重试

#### 已有基础

`frontend/src/app/workspace/knowledge/[kb_id]/page.tsx` 已有重试按钮（`reindexDocument`）。

#### 确认行为

- failed / stale 状态文档显示重试按钮。
- 点击后触发重新处理。
- 处理完成后状态变为 processing，然后 ready。

#### 验收

- 用户能主动触发失败文档的重新处理。
- 重试后状态正确更新。

## 七、权限与安全

### 7.1 记忆权限原则

1. 普通用户只能查看和修改自己的记忆。
2. Admin 可以查看所有用户的记忆（只读）。
3. Space owner 可以查看该空间内所有用户的记忆（只读）。
4. 记忆删除不可逆，需要 Toast 撤回机制。

### 7.2 引用来源展示原则

1. 只在 AI 回复确实引用了资料时显示 badge。
2. 引用列表不暴露文档的完整 URL 或内部 ID。
3. 用户没有权限的知识库文档不显示在引用中。

## 十一、风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 引用 metadata 需要改造 agent | 可能影响稳定性 | 先做前端展示，metadata 格式先固定不变 |
| Admin 记忆查看需要数据隔离 | 权限错误会导致越权访问 | 严格校验 user_id 和 role |
| 记忆大量时前端性能 | 记忆很多时加载慢 | 分页或虚拟列表 |
| 文档重试可能再次失败 | 用户困惑 | 显示失败原因 |

---

## 十二、版本结论

v1.4.6 不做复杂的知识图谱或记忆自动分析。

它只做三件事：

> **让用户看见 AI 用了什么资料。让用户看见自己记住了什么。让用户能纠正错误。**

这是 MicX 从"能用"到"可信"的关键一步。
