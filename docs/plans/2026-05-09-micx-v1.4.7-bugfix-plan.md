## 已修复的 Bug

### 1. 空格内容不应发送 ✅
**问题**: 用户输入只有空格时，按回车也会发送消息
**根因**: `prompt-input.tsx` 中 `onSubmit` 调用前没有检查空白内容
**修复位置**: `frontend/src/components/ai-elements/prompt-input.tsx`
**修复方案**:
```typescript
// 在 handleSubmit 的 .then() 中添加空白检查
if (!text || text.trim().length === 0) {
  return;
}
```
**状态**: ✅ 已修复

### 2. 导出 PDF 点取消仍弹窗 ✅
**问题**: 导出为 PDF 时，点击打印对话框的取消后，仍然显示"导出成功" toast
**根因**: `toast.success()` 在 switch 语句后无条件调用
**修复位置**: `frontend/src/components/workspace/export-trigger.tsx`
**修复方案**:
```typescript
case "pdf":
  exportThreadAsPDF(agentThread, messages);
  return;  // PDF 使用 window.print()，无法检测用户是否真正打印
```
**状态**: ✅ 已修复

### 3. 复制功能不生效 ✅
**问题**: 点击复制按钮后，复制操作可能失败但没有任何反馈
**根因**: `navigator.clipboard.writeText()` 没有 await 和错误处理
**修复位置**: `frontend/src/components/workspace/copy-button.tsx`
**修复方案**:
```typescript
const handleCopy = useCallback(async () => {
  try {
    await navigator.clipboard.writeText(clipboardData);
    setCopied(true);
    setTimeout(() => setCopied(false), 2000);
  } catch (err) {
    console.error("Failed to copy:", err);
  }
}, [clipboardData]);
```
**状态**: ✅ 已修复

### 4. 新对话/历史对话切换后回车导致聊天窗口重置 ✅
**问题**: 在新对话和历史对话之间切换后，输入内容按回车，聊天窗口被重置
**根因**: InputBox 的 key 策略不当。当 `isNewThread` 从 `true` 变为 `false` 时，key 变化导致组件强制卸载并重新挂载
**修复位置**: `frontend/src/app/workspace/chats/[thread_id]/page.tsx`
**修复方案**:
```typescript
// 之前: key={isNewThread ? (initialPrompt ?? "new-thread") : threadId}
// 现在: key={threadId}
```
**状态**: ✅ 已修复

### 5. reasoning 模型选中态错误 ⚠️
**问题**: 选中 reasoning/thinking 模式后，再次打开下拉菜单，错误显示 gemma4 为选中状态
**根因**: mode 选择和 model 自动选择之间的交互问题。mode 改变时 effect 会自动选择支持该 mode 的 model，但 model 下拉框的选中态基于 `context.model_name`，可能与用户期望不符
**修复位置**: `frontend/src/components/workspace/input-box.tsx`
**分析状态**: ⚠️ 已分析但未修复

**详细分析**:
- `handleModeSelect` 调用 `onContextChange` 更新 mode
- 同时 `context.mode` 的变化触发了 auto-selection effect
- effect 会自动选择支持新 mode 的 model
- 但 model 下拉框显示的 `selectedModel?.display_name` 可能在用户选择后立即被 effect 改变

**建议修复方案**:
需要更细致地处理 mode 选择和 model 选择之间的同步。

**优先级**: 中
**问题**: 镜像体积 4.5G，远大于万象服务的 200M，导致发布时间超过 3 小时
**根因**: 分析中
**需要检查**:

- dev target 包含编译工具链
- 可能的缓存问题

**初步分析**:
```dockerfile
# backend/Dockerfile
FROM python:3.12-slim-bookworm AS builder  # 基础镜像
# Stage 2 (dev) 包含 build-essential, nodejs, docker CLI
# Stage 3 (runtime) 才是生产镜像
```

**建议优化**:
1. 确保使用 production target 构建
2. 检查 docker-compose 中是否使用了正确的 target
3. 考虑使用更小的基础镜像
4. 优化依赖安装
