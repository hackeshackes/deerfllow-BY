### Bug 6: 多次点击历史对话后自动跳转新对话 🔴

**优先级**: P0
**影响**: 用户体验严重破坏，无法正常使用历史对话

**根因分析**:
```typescript
// use-thread-chat.ts
useEffect(() => {
  if (pathname.endsWith("/new")) {
    setIsNewThread(true);
    setThreadId(uuid());
    return;
  }
  if (threadIdFromPath === "new") {
    return;  // ⚠️ 问题点: 快速切换时可能误判
  }
  setIsNewThread(false);
  setThreadId(threadIdFromPath);
}, [pathname, threadIdFromPath]);
```

**问题**:
1. React 状态更新是异步的，`threadIdFromPath` 可能还未更新
2. 当 URL 从 `/chats/123` 变为 `/chats/456` 时，`pathname` 变化先触发 effect
3. effect 读取的 `threadIdFromPath` 可能仍是旧值
4. 如果 `threadIdFromPath === "new"` 的判断在此时为 true，会 return 不更新状态

**修复方案**:

```typescript
// use-thread-chat.ts
// 方案: 始终信任 URL，移除中间状态
useEffect(() => {
  const isNew = pathname.endsWith("/new");

  if (isNew) {
    // 如果当前不是 new thread，创建一个新的
    if (!isNewThread || threadIdFromPath === "new") {
      setIsNewThread(true);
      setThreadId(uuid());
    }
    return;
  }

  // 信任 URL，永远不返回
  setIsNewThread(false);
  setThreadId(threadIdFromPath);
}, [pathname, threadIdFromPath]);

// ⚠️ 需要移除 isNewThread 的条件返回逻辑
```

**修改文件**:
- `frontend/src/components/workspace/chats/use-thread-chat.ts`

### Bug 8: 上传图片后不能正常分析 🔴

**优先级**: P0
**影响**: 核心图片理解功能

**根因分析**:
```typescript
// hooks.ts sendMessage
const filesForSubmit: FileInMessage[] = uploadedFileInfo.map((info) => ({
  filename: info.filename,
  size: info.size,
  path: info.virtual_path,  // ⚠️ 路径可能不正确
  status: "uploaded" as const,
}));

await thread.submit({
  messages: [{
    type: "human",
    content: [...],
    additional_kwargs: {
      files: filesForSubmit  // ⚠️ AI 可能无法访问 /mnt/ 路径
    }
  }]
});
```

**检查项**:
1. 文件是否正确上传到 `/mnt/user-data/{workspace}/uploads/`
2. `uploadFiles` API 是否正确返回 virtual_path
3. 后端是否正确将 virtual_path 转换为实际文件路径
4. Vision 模型是否正确接收到图片数据

**修复方向**:
- 检查 `backend/app/api/uploads.py` 文件上传逻辑
- 检查 `ViewImageMiddleware` 是否正确处理图片
- 检查 `additional_kwargs.files` 是否正确传递到 AI

### Bug 9: 附件功能是摆设 🔴

**优先级**: P0
**影响**: 核心文件上传功能

**根因分析**:
```typescript
// prompt-input.tsx handleSubmit
Promise.all(files.map(...))
  .then((convertedFiles) => {
    // ⚠️ 问题: text 为空时不应该继续
    if (!text || text.trim().length === 0) {
      return;  // 我们刚加的检查
    }
    const result = onSubmit({ text, files: convertedFiles }, event);
  });
```

**可能问题点**:
1. `AddAttachmentsButton` 点击事件未正确触发文件选择器
2. 文件选择后未正确添加到 `attachments.files` 状态
3. 文件数据未正确传递到 `sendMessage`

**检查流程**:
```
用户点击附件按钮
  → AddAttachmentsButton.onClick
  → attachments.openFileDialog()
  → 文件选择 → handleChange
  → attachments.add(files)
  → PromptInput 显示附件
  → 用户提交 → handleSubmit
  → message.files 包含附件
  → sendMessage(threadId, message)
```

**修复方案**:
1. 检查 `usePromptInputAttachments` 的 `add` 方法
2. 检查 `handleChange` 是否正确处理多文件
3. 检查 `message.files` 是否正确传递

### Bug 10: 其他模式选不了 🔴

**优先级**: P0
**影响**: 核心 mode 选择功能

**根因分析**:
```typescript
// input-box.tsx
const handleModeSelect = useCallback(
  (mode: InputMode) => {
    onContextChange?.({
      ...context,
      mode: getResolvedMode(mode, supportThinking),
      // ⚠️ reasoning_effort 可能未正确设置
      reasoning_effort: mode === "ultra" ? "high" : ...
    });
  },
  [onContextChange, context, supportThinking],
);
```

**可能问题**:
1. Mode Selector 的 `onSelect` 被 DropdownMenu 拦截
2. `getResolvedMode` 返回意外值
3. `context.mode` 更新后触发 auto-selection effect，覆盖用户选择

**修复方案**:
```typescript
// 添加调试日志确认问题点
const handleModeSelect = useCallback(
  (mode: InputMode) => {
    console.log("Mode selected:", mode, "supportThinking:", supportThinking);
    const resolved = getResolvedMode(mode, supportThinking);
    console.log("Resolved mode:", resolved);
    onContextChange?.({
      ...context,
      mode: resolved,
      reasoning_effort: mode === "ultra" ? "high" :
                       mode === "pro" ? "medium" :
                       mode === "thinking" ? "low" : "minimal",
    });
  },
  [onContextChange, context, supportThinking],
);
```

### Bug 13: reasoning 模型选中态错误 ⚠️

**优先级**: P1
**影响**: Mode 选择体验

**详细分析**:
```typescript
// input-box.tsx

// Model Selector 的选中态
{m.name === context.model_name ? (
  <CheckIcon className="ml-auto size-4" />
) : ...}

// 但 mode 改变时，effect 会自动选择 model
useEffect(() => {
  if (models.length === 0) return;

  const currentModel = models.find((m) => m.name === context.model_name);
  const fallbackModel = currentModel ?? models[0]!;
  const supportsThinking = fallbackModel.supports_thinking ?? false;
  const nextModelName = fallbackModel.name;
  const nextMode = getResolvedMode(context.mode, supportsThinking);

  if (context.model_name === nextModelName && context.mode === nextMode) {
    return;
  }

  // ⚠️ 这会自动改变 model，可能覆盖用户选择
  onContextChange?.({
    ...context,
    model_name: nextModelName,
    mode: nextMode,
  });
}, [context, models, onContextChange]);
```

**问题**: 当用户选择 thinking mode 时，effect 会自动选择一个支持 thinking 的 model。但如果在 model selector 打开时 effect 触发，model selector 的显示会被覆盖。

**修复方案**:
```typescript
// 方案: 添加一个 "userIsSelectingModel" 状态，effect 执行时跳过
const [userIsSelectingModel, setUserIsSelectingModel] = useState(false);

// ModelSelector 打开时设置
<ModelSelector
  open={modelDialogOpen}
  onOpenChange={(open) => {
    setModelDialogOpen(open);
    setUserIsSelectingModel(open);
  }}
>

// effect 中检查
useEffect(() => {
  if (userIsSelectingModel) {
    return;  // 用户正在选择，不自动覆盖
  }
  // ... 原有逻辑
}, [context, models, onContextChange, userIsSelectingModel]);
```

### Bug 17: Docker 镜像 4.5G 🛠️

**优先级**: P1
**影响**: 部署效率

**根因分析**:
```dockerfile
# backend/Dockerfile
FROM python:3.12-slim-bookworm AS builder  # ~150MB base
# Stage 2 (dev) 包含:
# - build-essential
# - nodejs (~200MB)
# - docker CLI
# - uv cache
# 最终 dev target 体积约 2-3GB

# Stage 3 (runtime) ~500MB
# 但万象服务使用的是 dev target！
```

**检查 docker-compose**:
```yaml
# docker-compose-dev.yaml
gateway:
  build:
    target: dev  # ⚠️ 使用 dev target！
```

**修复方案**:
1. **方案A**: 为万象服务使用 runtime target
```yaml
gateway:
  build:
    target: runtime  # 使用更小的 runtime 镜像
```

2. **方案B**: 优化 dev target 体积
```dockerfile
# dev target 中移除不必要的工具
RUN apt-get remove -y build-essential nodejs
# 只保留运行时必需的
```

3. **方案C**: 分离构建和运行环境
```dockerfile
# 只在需要时构建，运行时使用预编译的 wheel
```
