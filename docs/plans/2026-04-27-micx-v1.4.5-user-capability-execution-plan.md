## 一、实施原则

1. **不增加新建页复杂度**：ChatStartPanel 继续隐藏，不恢复大面积起手引导。
2. **动作发生在对话完成后**：沉淀入口优先放在对话页顶部工具区或消息工具栏。
3. **先做最短闭环**：总结、保存资料、生成草稿、跳转预填，优先于复杂编辑器。
4. **权限不放松**：私有、空间共享、全局边界沿用已有模型。
5. **后端只补必要 API**：不重构知识库、检索、工作流底层。
6. **可回滚**：每个 Phase 独立可验证，失败时可以只关闭对应入口。

## 六、Phase 3：保存文本到资料库

### 目标

让用户把总结或 AI 回复直接保存到资料库，不需要导出再上传。

### 后端新增

`backend/app/gateway/routers/knowledge.py`

新增接口：

```bash
POST /api/knowledge/{kb_id}/documents/text
```

请求体：

```json
{
  "title": "string",
  "content": "string",
  "source_type": "thread_summary | assistant_message | full_thread",
  "source_thread_id": "string",
  "source_message_id": "optional"
}
```

行为：

- 校验用户对知识库有写入权限。
- 将文本保存为 Markdown 文档。
- 进入现有文档处理流程。
- metadata 记录来源线程。

### 前端新增

`frontend/src/components/workspace/capture/save-to-knowledge-dialog.tsx`

职责：

- 加载可写知识库列表。
- 显示目标可见范围。
- 编辑标题。
- 保存并提示结果。

### 前端修改

`frontend/src/core/knowledge/api.ts`

新增：

```ts
saveTextDocument(kbId, payload)
```

### 验收

- 可将线程总结保存为资料库文档。
- 保存后资料详情可见该文档。
- 没有权限时后端返回明确错误。
- 保存到共享空间前弹窗明确提示谁可见。

### 验证

```bash
cd frontend && pnpm lint && pnpm typecheck
cd backend && make lint
```

## 七、Phase 4：资料质量状态增强

### 目标

资料库页面显示文档是否可用，减少用户不知道资料是否生效的问题。

### 前端修改

`frontend/src/app/workspace/knowledge/[kb_id]/page.tsx`

显示字段：

- `status`
- `chunk_count`
- `token_count`
- `processed_at`

状态 badge：

| status | badge |
|--------|-------|
| processing | 处理中 |
| ready | 可用 |
| failed | 失败 |
| stale | 可能过期 |

### 验收

- 上传或保存后的文档状态可见。
- 失败状态不静默。
- 状态横向排列，不影响标题可读性。

## 十三、开发顺序建议

优先级从高到低：

1. Phase 1 + Phase 2：先让“沉淀入口 + 总结草稿”可见。
2. Phase 3：打通保存到资料库，这是最核心闭环。
3. Phase 4：资料状态增强，降低不信任感。
4. Phase 6：自动化预填，复用现有任务 API。
5. Phase 5：工作流草稿，如果现有 skills API 适配成本较高，可降级为复制草稿。
6. Phase 7：所有共享动作统一确认。

最小可交付版本：

- 沉淀入口
- 总结草稿
- 保存到资料库
- 资料状态显示

完整版本：

- 最小可交付版本
- 生成工作流草稿
- 生成自动化草稿
- 共享确认统一化

## 十四、需要用户确认的产品取舍

推荐默认选择完整版本。

### 方案 A：完整 v1.4.5

包含沉淀、保存资料、资料状态、工作流草稿、自动化草稿、共享确认。

优点：真正完成“沉淀与复用”闭环。  
缺点：后端可能需要新增文本保存接口和工作流草稿保存能力。

### 方案 B：轻量 v1.4.5

只做沉淀、总结、保存资料、资料状态。

优点：交付快，风险低。  
缺点：复用和自动化闭环不完整。

### 方案 C：只做 UI 原型

只做前端入口和草稿弹窗，不打通后端保存。

优点：最快看到界面。  
缺点：用户不能真正使用，不建议。

推荐：**方案 A**。如果开发中发现工作流 API 成本过高，可把工作流保存降级为“生成草稿 + 复制”，但不要砍掉资料保存闭环。
