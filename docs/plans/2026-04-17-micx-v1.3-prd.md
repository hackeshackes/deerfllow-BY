## 一、版本目标与核心价值

### 1.1 版本愿景

v1.3 将 MicX 从**即时AI助手**进化为**可持续运转的AI工作平台**：

| 现有能力 | v1.3 新增 | 价值跃升 |
|---------|----------|---------|
| 手动触发单次任务 | **定时/周期任务** | 从"需要时调用"到"自动持续运转" |
| 上传文件做上下文 | **RAG 语义检索** | 从"文件匹配"到"智能理解" |
| AI结果仅展示 | **结果推送/存档** | 从"被动查看"到"主动通知" |

### 1.2 核心场景

**场景A - 定时任务**
```
用户设定: "每天早上9点给我发送科技新闻摘要"
系统行为: 每日cron触发 → deep-research skill → 生成摘要 → 推送至用户消息列表
```

**场景B - RAG知识问答**
```
用户上传: 10份内部文档（PDF/Word/TXT）
用户提问: "在我们Q3文档里，关于XX项目的进展是什么？"
系统行为: 语义检索 → 找到相关段落 → 基于上下文生成准确回答
```

**场景C - 组合：定时+RAG**
```
用户设定: "每周一早上10点，基于我上传的行业报告，自动生成竞争分析"
系统行为: 定时触发 → RAG检索相关文档 → deep-research补充外部信息 → 生成报告
```

## 二、功能需求详述

### 2.1 定时任务系统

#### 2.1.1 任务创建

**用户操作入口**:
- 输入框指令: `/schedule [任务描述]` 或自然语言描述意图
- UI入口: 新建任务按钮 → 任务创建面板

**任务定义结构**:
```yaml
task:
  name: string                    # 任务名称（用户可见）
  description: string             # 任务描述
  trigger:
    type: cron | interval | one_time
    config:                       # 触发器配置
      cron: "0 9 * * *"         # 标准cron表达式
      timezone: "Asia/Shanghai"  # 时区
      start_date: datetime       # 可选：开始日期
      end_date: datetime         # 可选：结束日期
  prompt: string                  # 任务执行的prompt模板
  model: string                   # 使用的模型（可选，默认继承）
  skills: string[]                # 需要加载的技能（可选）
  notification:
    enabled: boolean
    channels: [in_app, email, webhook]  # 通知渠道
  output:
    save_to_thread: boolean       # 是否保存执行结果到线程
    webhook_url: string           # Webhook推送地址（可选）
```

#### 2.1.2 任务管理

**任务列表 UI** (`/workspace/tasks`):
- 我的任务列表（创建的任务）
- 任务状态：活跃/暂停/已完成
- 下次执行时间
- 执行历史（最近10次）

**任务操作**:
- 启用/暂停
- 立即执行（手动触发）
- 编辑
- 删除
- 查看执行日志

#### 2.1.3 任务执行

**执行流程**:
1. Scheduler 监听 cron 表达式
2. 到期触发 → 创建新线程（thread_type: "scheduled"）
3. 使用 prompt 模板创建初始消息
4. 加载指定 skills
5. 执行 agent 流程
6. 结果保存到线程
7. 发送通知（如果启用）

**执行上下文**:
- 定时任务创建独立的 thread
- 定时任务可访问用户的 RAG 知识库
- 支持继承创建任务时的上下文快照

#### 2.1.4 任务执行历史

**历史记录包含**:
- 执行时间
- 执行时长
- 状态：成功/失败/部分成功
- 生成的 artifact 列表
- Token 消耗（计入用户统计）

### 2.2 RAG 知识增强系统

#### 2.2.1 知识库管理

**用户操作入口**:
- `/workspace/knowledge` 或 设置页面入口

**知识库结构**:
```yaml
knowledge_base:
  id: uuid
  name: string                    # 知识库名称
  description: string             # 描述
  documents:                      # 文档列表
    - id: uuid
      name: string
      type: pdf | docx | txt | md | csv
      size: int                  # bytes
      uploaded_at: datetime
      chunk_count: int           # 分块数量
      status: processing | ready | error
  embedding_model: string         # 嵌入模型（可配置）
  created_at: datetime
  updated_at: datetime
```

**文档上传**:
- 支持格式: PDF, DOCX, TXT, MD, CSV
- 单文件限制: 50MB
- 自动分块: ~500 tokens/块，重叠50 tokens
- 进度显示: 上传 → 解析 → 分块 → 向量化

**文档管理**:
- 删除文档（同时删除向量）
- 重新索引
- 查看文档解析状态

#### 2.2.2 语义检索

**检索 API**:
```python
async def semantic_search(
    query: str,
    knowledge_base_id: uuid,
    top_k: int = 5,
    similarity_threshold: float = 0.7
) -> list[SearchResult]:
```

**SearchResult 结构**:
```python
class SearchResult:
    document_id: uuid
    document_name: string
    chunk_content: string        # 原文片段
    chunk_metadata: {           # 元信息
        page: int,
        section: string
    }
    similarity_score: float
```

#### 2.2.3 RAG 集成到 Agent

**Agent 调用 RAG 的时机**:
1. 用户上传文档并提问 → 自动检索
2. 用户明确指令: "在我上传的文档里找..."
3. 系统自动判断：用户提问可能与已上传文档相关

**RAG 上下文注入**:
```
<context from knowledge base: "文档名.pdf">
<chunk 1> ... 相关内容 ...
<chunk 2> ... 相关内容 ...
</context>
```

**多知识库支持**:
- 用户可创建多个知识库（个人/工作/项目）
- 提问时可选择检索范围：全部/指定知识库

### 2.3 Webhook 与通知

#### 2.3.1 Webhook 配置

**系统级 Webhook** (Admin 配置):
- 定时任务完成通知
- RAG 文档处理完成通知
- 系统告警

**用户级 Webhook** (用户配置):
- 个人任务的 webhook 推送
- API token 管理

#### 2.3.2 通知渠道

| 渠道 | 说明 | 优先级 |
|------|------|--------|
| In-App | 消息列表通知 | P0 |
| Email | 邮件通知 | P1 |
| Webhook | 推送到外部系统 | P1 |

## 三、技术方案

### 3.1 定时任务系统

**技术选型**:
- 后台调度: APScheduler（成熟稳定，支持 cron/interval/date）
- 任务存储: SQLite（复用现有 checkpointer 存储）
- 任务队列: APScheduler 内置 + 可扩展为 Celery（当前规模 SQLite 足够）

**数据库模型**:
```python
class ScheduledTask(BaseModel):
    id: uuid
    user_id: uuid
    name: str
    description: str
    trigger_type: str  # cron, interval, one_time
    trigger_config: dict  # JSON存储cron等配置
    prompt_template: str
    model_name: str | None
    skill_names: list[str]
    notification_config: dict
    output_config: dict
    status: str  # active, paused, completed
    created_at: datetime
    updated_at: datetime

class TaskExecution(BaseModel):
    id: uuid
    task_id: uuid
    thread_id: str | None  # 关联的线程
    status: str  # pending, running, success, failed
    started_at: datetime
    completed_at: datetime | None
    result_summary: str | None
    error_message: str | None
    token_used: int
```

**API 端点**:
```
POST   /api/tasks                    # 创建任务
GET    /api/tasks                    # 列表任务
GET    /api/tasks/{id}               # 获取任务详情
PUT    /api/tasks/{id}               # 更新任务
DELETE /api/tasks/{id}               # 删除任务
POST   /api/tasks/{id}/run           # 手动执行
POST   /api/tasks/{id}/pause         # 暂停
POST   /api/tasks/{id}/resume        # 恢复
GET    /api/tasks/{id}/executions   # 执行历史
```

**前端页面**:
- `/workspace/tasks` - 任务列表
- `/workspace/tasks/new` - 创建任务
- `/workspace/tasks/{id}` - 任务详情/编辑

### 3.2 RAG 知识增强系统

**技术选型**:
- 向量存储: ChromaDB（轻量、易部署、支持本地）
- 嵌入模型: sentence-transformers (all-MiniLM-L6-v2) 或 OpenAI text-embedding-3-small
- 文档解析: PyMuPDF(PDF), python-docx(DOCX), pandas(CSV)
- LangChain: 用于 RAG pipeline

**数据库模型**:
```python
class KnowledgeBase(BaseModel):
    id: uuid
    user_id: uuid
    name: str
    description: str
    embedding_model: str  # 嵌入模型名称
    created_at: datetime
    updated_at: datetime

class Document(BaseModel):
    id: uuid
    knowledge_base_id: uuid
    original_name: str
    file_type: str
    file_size: int
    storage_path: str  # 文件实际存储路径
    status: str  # processing, ready, error
    chunk_count: int
    token_count: int
    uploaded_at: datetime
    processed_at: datetime | None

class DocumentChunk(BaseModel):
    id: uuid
    document_id: uuid
    chunk_index: int
    content: str
    token_count: int
    embedding_id: str  # ChromaDB 中的 ID
    metadata: dict  # 页码、位置等
```

**API 端点**:
```
# 知识库管理
POST   /api/knowledge              # 创建知识库
GET    /api/knowledge              # 列表知识库
GET    /api/knowledge/{id}         # 获取知识库详情
PUT    /api/knowledge/{id}         # 更新知识库
DELETE /api/knowledge/{id}         # 删除知识库

# 文档管理
POST   /api/knowledge/{id}/documents     # 上传文档
GET    /api/knowledge/{id}/documents     # 列表文档
DELETE /api/knowledge/{id}/documents/{doc_id}  # 删除文档
POST   /api/knowledge/{id}/documents/{doc_id}/reindex  # 重新索引

# 检索
POST   /api/knowledge/{id}/search       # 语义检索
```

**向量数据库**:
- ChromaDB 使用 SQLite 后端，存储在 `data/chroma/` 目录
- 每个用户独立的 collection: `kb_{knowledge_base_id}`
- 支持增量更新和删除

**RAG Pipeline**:
```
用户提问
    ↓
判断是否需要 RAG（用户指定 / 自动检测）
    ↓
语义检索 → Top-K 相关片段
    ↓
构建上下文 → <context> + 原始提问
    ↓
注入 Agent 的 messages
    ↓
生成回答（带引用标注）
```

## 四、非功能性需求

### 4.1 性能

| 指标 | 目标 |
|------|------|
| 定时任务调度延迟 | < 1分钟 |
| RAG 检索延迟 | < 2秒（1000文档规模） |
| 文档上传 + 处理 | < 30秒（10MB PDF） |

### 4.2 限制

| 资源 | 限制 |
|------|------|
| 单用户最大知识库数 | 10 |
| 单知识库最大文档数 | 100 |
| 单文档最大大小 | 50MB |
| 单用户最大定时任务数 | 50 |
| 定时任务最小间隔 | 1分钟 |

### 4.3 安全

- 知识库仅创建者可见（工作区共享待后续版本）
- Webhook URL 需 HTTPS
- 文档内容仅用于 RAG，不用于模型训练

## 六、验收标准

### 6.1 定时任务系统

| 用例 | 验收条件 |
|------|---------|
| 创建 cron 任务 | 输入 cron 表达式，系统正确计算下次执行时间 |
| 定时执行 | 到达触发时间，任务自动执行，结果保存到线程 |
| 手动执行 | 点击"立即执行"，任务立即运行 |
| 暂停/恢复 | 暂停后 cron 不触发，恢复后继续 |
| 执行历史 | 显示所有执行记录，包含状态和摘要 |

### 6.2 RAG 知识增强

| 用例 | 验收条件 |
|------|---------|
| 创建知识库 | 填写名称描述，成功创建 |
| 上传 PDF | 文件正确解析，内容可检索 |
| 语义检索 | 输入问题，返回相关文档片段 |
| RAG 问答 | 上传文档后提问，AI 回答基于文档内容 |
| 删除文档 | 向量同步删除，知识库状态更新 |

### 6.3 集成

| 用例 | 验收条件 |
|------|---------|
| 定时任务使用 RAG | 定时任务可加载用户的知识库 |
| 任务通知 | 任务完成，消息列表出现通知 |
| 前端无报错 | 各页面正常加载，无 console error |
