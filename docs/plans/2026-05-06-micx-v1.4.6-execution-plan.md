## 一、实施原则

1. **不新增一级导航**：引用来源在对话内展示，记忆中心复用设置页，Admin 记忆查看复用 admin 用户管理。
2. **先确认 metadata 格式**：引用来源需要后端在 agent 响应中注入，先确认格式再开发前端展示。
3. **权限严格校验**：Admin 记忆查看涉及数据隔离，必须在后端做好权限校验。
4. **可回滚**：每个 Phase 独立验证，失败时可只关闭对应功能。

## 二、当前代码基础

### 2.1 AI 消息与引用来源

关键文件：

- `frontend/src/components/workspace/messages/` — AI 消息渲染组件
- `backend/app/gateway/routers/threads.py` — thread 相关 API

已有能力：

- AI 消息通过 LangGraph streaming 返回，已有的消息渲染组件。

缺口：

- Agent 响应 metadata 中没有 citations 字段。

### 2.2 用户记忆

关键文件：

- `frontend/src/components/workspace/settings/memory-settings-page.tsx` — 已有完整记忆设置页
- `frontend/src/core/memory/hooks.ts` — 记忆相关 React hooks

已有能力：

## 四、Phase 1：确认 Agent citations 格式

### 目标

确认后端 agent 响应 metadata 中的 citations 格式，前端才能开发展示组件。

### 后端确认

查看当前 agent 响应结构：

1. 在 `backend/packages/harness/deerflow/agents/` 中搜索是否有 citations 或 source_documents 相关字段。
2. 确认 knowledge retrieval 返回的数据结构。
3. 确定 citations metadata 的注入位置。

如果当前 agent 没有 citations 字段：

- 评估是否可以在知识检索工具返回结果时注入 metadata。
- 评估是否可以在前端通过分析消息内容来模拟引用（不推荐，精度不够）。

### 验收

- 确认 citations 格式（字段名、数据结构）。
- 如果无法从后端获取，明确告知用户并降级为"仅显示文档名称"（通过 search API 匹配）。

## 五、Phase 2：AI 消息引用来源展示

### 目标

在 AI 消息组件中显示引用来源 badge。

### 前端新增

`frontend/src/components/workspace/messages/citation-badge.tsx`

职责：

- 接收 citations 数据（doc_name, kb_id, doc_id, status, similarity_score）。
- 显示 badge：`📄 引用了 N 份资料`。
- 点击展开文档列表。
- 文档列表显示名称 + 状态 badge。

### 前端修改

在 AI 消息渲染组件中：

1. 从消息 metadata 中提取 citations。
2. 如果有 citations，渲染 `CitationBadge` 组件。
3. 放置位置：AI 消息右下角或消息底部。

### 验收

- 有引用时显示 badge。
- 无引用时不显示空 badge。
- 文档列表可展开/收起。
- 点击文档名可跳转知识库详情页（如果用户有权限）。

## 七、Phase 4：Admin 记忆查看

### 目标

让 Admin 和 Space owner 能查看空间内用户的记忆（只读）。

### 后端新增

`backend/app/gateway/routers/memory.py`

新增 endpoint：

```bash
GET /api/admin/memory/{user_id}
```

校验：

1. 当前用户必须是 admin 或 space owner。
2. 如果是 space owner，必须确保目标用户在当前 space 内。
3. 返回的记忆数据是只读的，API 不提供任何写入操作。

### 前端新增

方案 A：在 admin 用户管理页面添加 "查看记忆" 按钮。

方案 B：新建 `frontend/src/app/workspace/admin/users/[user_id]/memory/page.tsx`。

推荐方案 A，复用现有 admin 用户管理 UI。

### 验收

- Admin 可以查看所有用户的记忆。
- Space owner 可以查看空间内用户的记忆。
- 普通用户不能访问该 API。
- 页面没有编辑/删除入口（只读）。

## 十三、技术债务说明

### Citations metadata 的不确定性

当前不确定 agent 响应中是否已有 citations 字段。如果需要在后端注入，需要：

1. 找到 knowledge retrieval 的调用位置。
2. 在返回结果中附加 metadata。
3. 确保 metadata 能传递到前端消息渲染组件。

如果无法在后端实现，前端降级方案：

- 在消息渲染时，检测是否有 `source_documents` 或类似字段。
- 如果没有，通过 `GET /api/knowledge/{kb_id}/search` 反向匹配（性能差，不推荐）。

### 记忆数据隔离

`get_memory_data()` 当前不需要参数（使用全局 context 中的 user_id）。

改造为支持指定 user_id 需要：

1. 修改 `memory/updater.py` 中的函数签名，增加 `user_id` 参数。
2. 修改 `memory/storage.py` 中的读取逻辑，支持按 user_id 读取。

这与 v1.4.4 的 memory path fix 模式相同，可以参考。
