# v1.6.0 启动会议议程

> **日期**:建议 2026-07-06(v1.5.5 全量上线后 +1 周)

> **参与人**:Tech Lead(主持)、Product Owner、AI 工程师 × 3、平台工程师 × 1、集成工程师 × 1、前端工程师 × 3、SRE
> **目标**:对齐 v1.6.0 范围、3 条线 owner、关键风险;6-7 周计划能否在 v1.5.5 经验基础上做对

参考:
- 规划:`docs/superpowers/plans/2026-06-22-micx-v1.6.0-multitenant-canvas.md`
- 路线图:`docs/superpowers/specs/2026-06-22-micx-ai-portal-roadmap-design.md` §4.3
- 前置:v1.5.5 灰度完成 + 3 部门回访

## 议程

### 1. 开场(5 min)— Tech Lead

- 1 句话回顾 v1.5.5 战果(36 commits / 240 tests / 36 commits / 36 commits,3 周灰度)
- 1 句话定调 v1.6.0:"从'能用'到'可运营'"
- 会议目标(对齐范围、认领 owner、识别风险)

### 2. v1.6.0 北星(5 min)— Product Owner

引用路线图 §4.3 的指标:
- DAU/员工数:25% → 40%
- 业务系统接入数:4 → 5+
- 自动化任务数:20 → 50
- 部门试点:3 → 5+(全公司)

> 重点:**这版本不是再加功能,是把 v1.5.5 的"能用"变成"能管"**

### 3. 三条线方案走查(30 min,每线 10 min)

每条线由 owner 介绍 5 分钟后,Q&A 5 分钟。

#### 3a. 多租户 / 资源治理线(owner:AI 工程师 + 平台)
- 资源隔离:workspace-scoped KB / skill / memory / connector
- 模型配额:Token/月、用户级 RPM、模型白名单
- 多模型路由:cost/quality/speed 三维度
- 成本看板:部门 / 项目 / 个人三级
- 可观测性:OpenTelemetry + Langfuse trace
- ABAC 基础(v1.5.4 推后的工作)
- **Q&A 焦点**:配额触发后的降级策略?谁来付超额的 Token?

#### 3b. AI 编排 / 画布线(owner:AI 工程师 + 前端)
- 画布 UI 骨架:节点 / 边 / 条件 / 并行
- 5 类节点:Agent / Tool / Prompt / Branch / Loop
- 保存 / 运行 / 版本
- 每次执行:trace + Token + 失败定位
- **Q&A 焦点**:非工程师能否配置?画布 vs SKILL.md 的关系?

#### 3c. 协作 / 连接器收尾线(owner:前端 + 集成)
- 共享对话评论(把 v1.5.5 没做的部分补上)
- Jira / Linear 工单连接器(P1)
- Slack 连接器(P2,若有余力)
- 跨部门发布 PublishService(workspace thread 推送给其他部门)
- **Q&A 焦点**:工单连接器是否会拉宽到 CMDB?Slack 必须吗?

### 4. v1.5.5 经验教训(15 min)— 全体

3 个 owner 各说 1 个"v1.5.5 我们做对了/做错了"。

模板:
- **做对了**:TDD 让 240 个测试在 PR 合并时直接绿;Post-* 阶段用 helper 消除了并行 worktree 冲突。
- **做错了**:SAML 推了 1 个版本导致 v1.5.4 范围漂移;连接器运行时先建后优化导致 DLQ 设计返工 1 次。
- **要尝试**:Pair rotation(每 2 周换线 owner 半天,避免知识孤岛)

### 5. 风险与依赖(10 min)— Tech Lead

- v1.5.5 灰度期间收集的 P2/P3 修复是否在 v1.6.0 范围?
- 可观测性 trace 数据是否能从 v1.5.5 的 audit 流迁移?
- Langfuse / OpenTelemetry 的 vendor 锁定风险?
- 画布 + 现有 SKILL.md 的兼容期策略?

### 6. 资源与排期(10 min)— Tech Lead + SRE

| 资源 | 数量 | 投入 |
|---|---|---|
| AI 工程师 | 3 | 治理线 2 + 编排线 1 |
| 平台工程师 | 1 | 治理线 |
| 前端工程师 | 3 | 编排线 2 + 协作线 1 |
| 集成工程师 | 1 | 协作线 |
| SRE | 1 | 全程 0.5 FTE |
| **合计** | 9 | **5+ 人团队等价** |

排期(6-7 周):
- Week 1-2:三线并行启动 + 公共基础设施
- Week 3-5:核心模块开发
- Week 6:集成 + 冒烟
- Week 7:内部灰度

### 7. 决策点(10 min)— 全体

需要今天决定的:
1. **ABAC 范围**:v1.5.4 推过来的 ABAC 是 v1.6.0 P0 还是 v1.7+?
2. **画布 vs SKILL.md**:画布是平级还是替代?
3. **Jira/Linear 是否合并需求**:两个连接器并行还是只做 1 个?
4. **跨部门发布**:发布是 copy-on-share 还是引用?
5. **v1.5.5 P2/P3 backlog**:并入 v1.6.0 还是单独 sprint?

### 8. 行动项(5 min)— Tech Lead

- [ ] 各线 owner 周二前在 `docs/plans/v1.6.0-line-{name}.md` 写下详细计划
- [ ] SRE 周三前评估 OpenTelemetry 后端选型(自建 vs SaaS)
- [ ] Product Owner 周三前回复"共享对话评论"优先级
- [ ] 周三 14:00 三线 kickoff 分会场
