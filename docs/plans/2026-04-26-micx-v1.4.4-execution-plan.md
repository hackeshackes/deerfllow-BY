**版本**: v1.4.4  
**对应 PRD**: `docs/plans/2026-04-26-micx-v1.4.4-prd-final.md`  
**日期**: 2026-04-26  
**状态**: 待用户确认后开始开发

---

## 一、实施原则

1. **先重组用户体验，再扩展能力**：尽量复用现有 backend 能力，避免大迁移。
2. **旧路由兼容**：`/workspace/tasks`、`/workspace/settings/skills`、`/workspace/knowledge` 不直接删除。
3. **权限不放松**：所有个人/共享边界继续使用已有 `owner_user_id`、`workspace_id`、`visibility`。
4. **小步验证**：每个 Phase 完成后运行相关检查，避免最后集中爆炸。
5. **上线后再测一遍**：本地测试通过后部署，部署后进行真实页面与 API 测试。

## 六、Phase 3：对话列表分区

### 目标

### 前端修改

#### 修改文件

`frontend/src/components/workspace/recent-chat-list.tsx`

新增分组逻辑：

```ts
const manualThreads = threads.filter(source === "manual" or missing source)
const automationThreads = threads.filter(source === "automation" or metadata.task_id)

```

分组标题：

每条对话显示：

- 更新时间

### 后端修改

#### 修改文件

`backend/app/gateway/routers/threads.py`

尽量复用现有 metadata filter。

新增 helper：

```python
def enrich_thread_metadata(metadata, user, workspace_id):
    metadata["space_type"] = "personal" if workspace_id == f"ws-{user.id}" else "workspace"
    metadata.setdefault("source", "manual")
```

写入点：

- create thread
- sync thread
- task-created thread
- channel-created thread（如果当前已有 channel metadata）

### 验收

## 十四、风险与应对

### 风险 1：改动范围偏大

应对：按 Phase 开发，每个 Phase 可独立验证。先做 IA 和前端包装，后做可选增强。

### 风险 2：旧任务/知识库数据兼容

应对：不删除旧字段，不做破坏性迁移。新增字段全部可选。

### 风险 3：用户不理解“工作流”和“技能”的关系

应对：普通用户只看到工作流，高级入口保留技能管理。

### 风险 4：新建对话资料来源没有真正进入 agent context

应对：v1.4.4 最低可先作为 metadata 和 prompt starter，后续版本深度接入 RAG context。
