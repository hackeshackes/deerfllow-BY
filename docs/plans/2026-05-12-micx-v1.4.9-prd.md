## 二、功能需求详述

### 2.1 新对话自动命名

#### 2.1.1 问题描述

当前新建对话时，对话标题统一显示为 "Untitled"，用户需要手动修改标题才能区分不同对话。这导致：

- 对话列表中大量 "Untitled"，难以识别
- 用户必须手动操作才能获得有意义的标题
- 记忆功能依赖标题做上下文，降低了自动化标题生成的价值

#### 2.1.2 解决方案

**策略**：对话创建后，首次 AI 回复完成时自动生成标题

**标题生成规则**：
1. **优先使用用户输入**：如果用户通过 URL 参数 `?prompt=` 预填了内容，使用输入内容前 30 字符
2. **AI 智能生成**：否则在首次 AI 回复完成后，调用标题生成逻辑
3. **兜底策略**：如果标题生成失败，保持 "Untitled"，用户可手动修改

**标题生成时机**：
- 用户发送第一条消息后
- AI 完成首次回复时触发
- 通过现有的 `TitleMiddleware` 机制实现

#### 2.1.3 数据模型

```python
# Thread 模型已有 title 字段，无需新增字段
class Thread(BaseModel):
    id: str
    title: str                    # 原字段，现支持自动生成
    workspace_id: str
    created_by: str
    # ...
```

#### 2.1.4 API 变更

```bash
# 新增：主动更新标题
PUT /api/threads/{id}/title
{
  "title": "用户反馈分析 - 2026年5月"
}

# 原有：创建线程时可选传入 title
POST /api/threads
{
  "workspace_id": "xxx",
  "title": "可选的初始标题"   # 新增可选字段
}
```

#### 2.1.5 前端变更

**1. 对话列表展示**
- 对话创建后立即显示占位标题 "新对话"（而非 "Untitled"）
- AI 首次回复完成后，标题更新为生成的内容
- 标题更新时无需刷新页面，通过 WebSocket 或轮询同步

**2. 新建对话交互**
- 点击「新建对话」时，对话列表立即出现新条目
- 标题显示为 "新对话..."（带省略号表示生成中）
- 生成完成后，省略号消失，显示真实标题

**3. 标题编辑**
- 用户可点击标题直接编辑
- 编辑后标题不再自动覆盖（用户优先原则）

#### 2.1.6 技术实现

**Backend**：
- 修改 `TitleMiddleware` 或创建 `AutoTitleMiddleware`
- 在首次消息交换完成后触发
- 支持从用户输入提取或调用 LLM 生成

**Frontend**：
- 对话列表添加标题状态管理
- 标题加载中状态 UI（"新对话..."）
- 标题编辑交互

### 2.3 技术债务清理

#### 2.3.1 异步并发优化

**问题**：当前代码中存在大量同步阻塞调用，在 IM 渠道多任务并发场景下性能较差。

**优化项**：

| 文件 | 当前代码 | 优化后 | 影响 |
|------|---------|--------|------|
| `task_tool.py` | `time.sleep(5)` | `asyncio.sleep()` | 轮询不再阻塞事件循环 |
| `local_sandbox.py` | `subprocess.run()` | `asyncio.create_subprocess_shell()` | 沙箱命令异步化 |
| `community/tools/*.py` (tavily, jina_ai, firecrawl) | `requests` | `httpx.AsyncClient` | 社区工具异步化 |
| `title_middleware.py` | `model.invoke()` | `model.ainvoke()` | 标题生成异步 |
| `memory updater` | `model.invoke()` | `model.ainvoke()` | 记忆更新异步 |

**实现要求**：
1. 保持功能完全一致，仅改异步
2. 添加适当的超时控制
3. 错误处理逻辑保持不变

#### 2.3.2 消息标准化修复

**问题**：`backend/app/gateway/services.py` 中消息标准化仅处理 `user` 角色，非 `user` 角色（`system`、`ai`、`tool`）的消息被错误归类。

**当前代码问题**：
```python
# TODO: handle other message types (system, ai, tool)
# 当前实现可能将 system/ai/tool 消息错误归类为 HumanMessage
```

**修复方案**：
```python
def normalize_message_type(message: dict) -> str:
    """标准化消息角色"""
    role = message.get("role", "")

    if role == "user":
        return "user"
    elif role == "system":
        return "system"
    elif role in ("assistant", "ai"):
        return "ai"
    elif role == "tool":
        return "tool"
    else:
        # 未知角色，记录日志并归类为 user
        logger.warning(f"Unknown message role: {role}")
        return "user"
```

**影响范围**：
- `/api/threads` 消息创建
- LangGraph checkpoint 序列化
- 记忆提取

**测试要求**：
- 验证 system/ai/tool 消息正确分类
- 验证 checkpoint 序列化/反序列化正确

## 四、验收标准

### 4.1 新对话自动命名

| 用例 | 验收条件 |
|------|---------|
| URL 预填对话 | 对话列表显示预填内容前 30 字符 |
| 普通对话 | 首次 AI 回复后，标题自动更新为生成内容 |
| 标题编辑 | 用户编辑后的标题不再被覆盖 |
| 加载状态 | 标题生成中显示 "新对话..." |
| 生成失败 | 保持 "新对话"，不显示错误 |

### 4.2 飞书群组工具

| 用例 | 验收条件 |
|------|---------|
| 列出群 | 返回用户所在的所有群列表 |
| 获取群详情 | 返回群名称、描述、创建者等信息 |
| 获取群成员 | 分页返回群成员列表 |
| 创建群 | 成功创建群并返回 chat_id |
| 错误处理 | 未配置飞书时返回友好错误提示 |

### 4.3 异步优化

| 用例 | 验收条件 |
|------|---------|
| 轮询不阻塞 | task_tool 轮询期间事件循环可处理其他任务 |
| 沙箱命令异步 | 长时间运行的 bash 命令不阻塞其他请求 |
| 社区工具异步 | tavily/jina_ai/firecrawl 调用不阻塞 |
| 标题生成异步 | 标题生成期间 UI 可响应 |
| 记忆更新异步 | 记忆更新不阻塞消息响应 |

### 4.4 消息标准化

| 用例 | 验收条件 |
|------|---------|
| user 消息 | 正确归类为 user |
| system 消息 | 正确归类为 system |
| ai 消息 | 正确归类为 ai |
| tool 消息 | 正确归类为 tool |
| 未知角色 | 记录日志并归类为 user，不崩溃 |
