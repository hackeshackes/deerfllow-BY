## Detail by scope

### 1. SQLite workflow persistence

The `WorkflowStore` and `VersionStore` Protocols now have a SQLite-backed
implementation, pluggable at process construction:

```bash
MICX_CANVAS_STORE=memory   # default (v1.6.0-canvas behaviour preserved)
MICX_CANVAS_STORE=sqlite   # drafts + versions survive gateway restarts
MICX_CANVAS_DB=...         # default .deer-flow/canvas.db; override for prod
```

Two tables: `workflows` and `workflow_versions`, each carrying a
JSON-snapshot of the full `Workflow` so new dataclass fields don't
need a migration. The companion `InMemoryWorkflowStore` /
`InMemoryVersionStore` gained a no-op `close()` so backend switching
doesn't require branching at the call site.

**Bug fix bonus**: v1.6.0-canvas shipped the router included but never
wired — production `POST /api/workflows` would 503. The lifespan now
calls `configure_canvas(...)` with the factory-resolved store pair.
Without this PR, SQLite persistence alone would not have been reachable.

### 2. workflow_id on quota records

`UsageRecord` carries an optional `workflow_id` (default `None`,
backward-compatible with pre-v1.6.1 callers like chat runs). The
canvas `/execute` route best-effort records `workflow_id` after the
executor returns. New API surface:

- `UsageTracker.tokens_in_window_for_workflow(...)` — per-workflow
  reverse-lookup so a quota audit can attribute consumption to a
  specific `Workflow`.
- `QuotaService.record_usage(...)` — single-record entry point used
  by the canvas router (so the post-execute accounting doesn't pay
  the cost of recomputing the same quota windows that the pre-check
  already computed).

### 3. Slack connector (HTTP Events variant)

`SlackConnector` implements `BaseConnector`. Mirrors the existing
Feishu / WeCom / DingTalk / Email connectors in
`app.gateway/connectors/`:

- **Outbound**: `POST https://slack.com/api/chat.postMessage` with
  bearer (`xoxb-...`) auth.
- **Inbound**: maps Slack `event_callback` `message` events to
  `ConnectorMessage` (skipping `reaction_added`, `bot_message`
  subtypes, etc. so the lead agent doesn't echo its own output back).
- **URL verification handshake**: a separate
  `verify_challenge_event()` helper echoes the Slack `challenge`
  so the bridge can short-circuit before delegating to
  `receive_webhook`.
- **Health**: `auth.test`.

Wired into `register_builtin_connectors(...)` so
`MICX_CONNECTORS={"slack": {"bot_token": "..."}}` lights it up
without an explicit adapter registration. YAML template ships at
`backend/app/gateway/connectors/slack/yaml/slack.example.yaml`.

**Explicit out of scope**: Socket Mode
(`apps.connections.open` + WebSocket subscription). The v1.6.x spec
flagged this as P2 — deferred to v1.6.2 / v1.7. HTTP Events are
sufficient for the spec'd P1 integration.

### 4. ABAC simplified scope

Landing the scope called for in the v1.6.0-canvas backlog
("ABAC simplified scope not landed"), per the v1.6.x design §3:

- `Subject` / `Resource` / `Action` value objects.
- `AttributePolicy` with a **fixed AST** of `equals`, `in`,
  `all_of`, `any_of`. **No `eval`/`exec`** — same audit story as
  the canvas `BranchNode`. First-matching-policy-wins traversal;
  no match → fail-closed deny.
- Two preset policies:
  - `OwnerOnlyPolicy` — `role == "owner"`.
  - `WorkspaceMemberPolicy` — owners pass through; members
    allowed only when `resource.workspace_id ∈ subject.workspaces`.

**Wire-up**: `POST /api/threads/{id}/publish` and
`POST /api/workflows/{id}/execute`. The latter fires the check
before the executor (and before the 503) so a missing-executor
config doesn't leak through authorization.

**Explicit out of scope**: full XACML-style evaluator + policies
file + UI editor + universal route coverage. The v1.6.x design
flags this as P2 (defer to v1.7+); RBAC v1.5.4 still applies for
coarse-grained owner / admin checks.

### 5. Playwright contract e2e for v1.6.x canvas

`e2e/tests/v1.6.x-canvas.spec.ts` — 4 contract tests against the
gateway under `E2E_LIVE=1`:

1. `GET /api/workflows` list envelope.
2. `POST /api/workflows` returns 200 + auto-commits a version.
3. `POST /api/workflows` with no name → 401 / 403 / 422.
4. `GET /api/workflows/{id}/versions` returns version-shaped entries.

Mirrors `v1.5.7-stability.spec.ts`. Default-skipped, so CI cost
stays at zero when not running the e2e lane.

## How to deploy

### Local (this repo's `docker/`)

```bash
cd /Users/baoyu/Documents/GitHub/deerfllow-BY
git fetch origin
git checkout v1.6.1-canvas-released   # or: git pull origin fix/v1.6.1-canvas-sqlite
docker compose -f docker/docker-compose.yaml up -d --build
```

The gateway image picks up the new `configure_canvas(...)` call in
lifespan automatically; restart picks up sqlite-store + ABAC +
Slack.

### To enable SQLite workflow persistence

```bash
MICX_CANVAS_STORE=sqlite
MICX_CANVAS_DB=/var/lib/deer-flow/canvas.db   # optional; default under .deer-flow/
```

### To enable the Slack connector

```bash
MICX_CONNECTORS=$(cat <<EOF
{
  "slack": {"bot_token": "${SLACK_BOT_TOKEN}"}
}
EOF
)
```

Or in `config.yaml` equivalent of `extensions_config.json`.

### Database migrations

None. New tables (`workflows`, `workflow_versions`) are created on
first call via `CREATE TABLE IF NOT EXISTS`.

### Production deploy

Pin frontend and gateway deploys to `v1.6.1-canvas-released`.

## Known limitations / v1.6.2 backlog

The v1.6.1 backlog is fully closed. Remaining forward work is
explicitly v1.6.2+ and is not gating:

- **Socket Mode for Slack** (`apps.connections.open` + WebSocket
  subscription) — much larger surface than HTTP Events (TLS,
  reconnect, ack handling).
- **Full ABAC** (XACML evaluator + policies file + UI editor +
  universal route coverage) — the v1.6.x design flags this as v1.7+.
- **`workflow_executions` table** — execution-history persistence
  in `MICX_CANVAS_STORE=sqlite` is deferred until the runtime
  accounts more than per-token sums.
- **D1 / D2 rollout gates** — internal → department → 3-department
  canary + P95 < 2s benchmark. This is process / ops work; not a
  code change.
- **Signing-secret verification on Slack webhooks** — current
  bridge passes the body to `receive_webhook` after the URL
  verification handshake, but the HMAC v0 signature on real events
  is not yet validated. Add before exposing a public webhook URL.
