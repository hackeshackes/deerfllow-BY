## Endpoints

All three require `require_owner_user(request)`. Non-owner → `403
仅拥有者可执行此操作`; unauthenticated → `401 需要先登录`.

### `POST /api/admin/secrets/upsert`

Create or replace a vault entry. `value=null` deletes the entry.

```json
{ "key": "models/dspark-v1.1-mida-brikie/api_key", "value": "sk-…" }
```

Returns:

```json
{
  "key": "models/dspark-v1.1-mida-brikie/api_key",
  "reference": "secret://models/dspark-v1.1-mida-brikie/api_key",
  "status": "upserted",
  "masked": true
}
```

Audit: `admin_secret.upserted` / `admin_secret.deleted` (the latter only
when the key was actually present before the delete).

### `POST /api/admin/secrets/rotate`

Atomic rotation of an env-routable secret **and** its on-disk vault
cipher. Requires `current_admin_password` to prevent an attacker who
hijacked the cookie from rotating `BETTER_AUTH_SECRET` and locking the
real owner out.

```json
{
  "key": "MICX_ADMIN_SECRET_KEY",
  "new_value": "…",
  "current_admin_password": "…"
}
```

Returns `cascades` and `cookie_invalidation` so the caller knows
whether to force a re-login (`session_hmac` in cascades).

Order of operations under `_secret_lock`:

1. Snapshot current cipher env value.
2. Write new value to `os.environ`.
3. Re-encrypt the on-disk vault under the new cipher (`tempfile.mkstemp` + `Path.replace`).
4. Re-upsert the key as a vault entry.

Rate limited at 10 requests / 60 s per client IP, sliding window. Over
limit returns `429` with `Retry-After` header.

### `GET /api/admin/secrets/status`

Classify every catalog secret:

```json
{
  "items": [
    {
      "key": "models/dspark-v1.1-mida-brikie/api_key",
      "state": "fresh",        // missing | placeholder | fresh | configured
      "is_placeholder": false,
      "last_rotated_at": "2026-07-16T10:00:00+00:00",
      "source": "vault",        // vault | env | seed_default
      "masked_value": "sk••••••••••••••••90"
    }
  ],
  "known_keys": ["…"],
  "vault_mtime": "2026-07-16T10:00:00+00:00"
}
```

By default only `KNOWN_VAULT_KEYS` (the operator-visible set) plus the
two privileged keys (`BETTER_AUTH_SECRET`, `MICX_ADMIN_SECRET_KEY`).
Pass `?include_all=true` to surface every `KNOWN_SECRET_KEYS` entry
(env-only providers like `OPENAI_API_KEY`, `TAVILY_API_KEY`, …).

## Known limitations / future work

- **Multi-process rotation** — current `threading.Lock` plus on-disk
  vault is single-process safe. If you run multiple gateway replicas
  pointing at the same `MICX_ADMIN_SECRET_KEY`, only one rotate per
  vault will be coherent; the others would race the cipher swap. M2
  should either pick a leader or move to a file lock.
- **`SECRETS_VAULT_ROUTABLE` enforcement** — `rotate` accepts any
  catalog key today. M2 should pin which keys are rotatable through
  the endpoint vs. require a manual `edit .env` flow.
- **Audit query surface** — events are written to
  `backend/.deer-flow/admin/audit.jsonl` but not yet surfaced via
  `/api/admin/audit` (the identity audit router covers the schema
  fields, but doesn't include `admin_secret.*` actions in its filter
  yet).
- **Frontend admin UI** — the three endpoints are ready; the admin
  console does not yet render them. Frontend work tracked separately.
